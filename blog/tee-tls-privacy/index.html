TLS and TEEs: How Cocos AI provides privacy and authentication

Danko Miladinovic, Jovan Djukic
February 19, 2026 · 4 min read

Introduction

Cocos AI is an advanced platform that leverages Confidential Computing and Trusted Execution Environments (TEEs) to enable secure multiparty computation (SMPC). Within its architecture, the Agent operates inside a TEE and plays a crucial role in establishing an attested TLS (aTLS) connection with the CLI, ensuring both secure communication and runtime integrity.

Implementation

COCOS AI adopts the RATS background-check model for remote attestation and implements an Intra-handshake attested TLS. A secure TLS channel is established every time a command is issued via Cosos AI CLI tool. The following diagram explains the inner workings.

As you can see the flow has a few differences compared to standard TLS.

Firstly, to ensure a fresh attestation report, CLI tool sends a nonce to the Agent. Cosos uses the Server Name Indication (SNI) extension for this purpose. The SNI is chosen because no servers are running in the CVM besides the Agent.

Secondly, the agent genearates a new ephemeral key pair for each connection regardless if that connection is with the same CLI user. During certificate creation, the attestation report is embedded directly into the X.509 certificate. By embedding the attestation report into the X.509 certificate, the TLS handshake remains unchanged. It is the job of the CLI user (the relaying party) to verify the attestation report extracted from the TLS certificate. Because the attestation report is bound to the X.509 certificate and the Agent is the only one with the private key of the X.509 certificate, the CLI user has enough proof that it is talking to the Agent.

The Agent also verifies the CLI certificate, ensuring that only members of the Computation are allowed to communicate with the Agent. The public keys of the Computation members are part of the attestation report. Thus, the same members can check who can communicate with the Agent.

The following Go snippet demonstrates how the certifiacate is generated.

    attestationData, err := p.attestationProvider.Attest(pubKeyDER, nonce)
	if err != nil {
		return nil, fmt.Errorf("failed to get attestation: %w", err)
	}

	extension := pkix.Extension{
		Id:    p.attestationProvider.OID(),
		Value: attestationData,
	}

    [...]

    csrMetadata := certs.CSRMetadata{
		Organization:    []string{p.subject.Organization},
		Country:         []string{p.subject.Country},
		CommonName:      p.subject.CommonName,
		Province:        []string{p.subject.Province},
		Locality:        []string{p.subject.Locality},
		StreetAddress:   []string{p.subject.StreetAddress},
		PostalCode:      []string{p.subject.PostalCode},
		ExtraExtensions: []pkix.Extension{extension},
	}

	csr, sdkerr := p.certsSDK.CreateCSR(ctx, csrMetadata, privateKey)
	if sdkerr != nil {
		return nil, fmt.Errorf("failed to create CSR: %w", sdkerr)
	}

	cert, err := p.certsSDK.IssueFromCSRInternal(ctx, p.cvmID, p.ttl.String(), string(csr.CSR), p.agentToken)
	if err != nil {
		return nil, err
	}

The behavior of the Attest() function depends on the underlying platform, such as AMD SEV-SNP or Intel TDX. Regardless of the platform, the attestation report always includes the Agents public ephemeral key and the nonce generated by the CLI. Once the certificate is obtained, a standard TLS connection can be established. The key difference compared to a regular TLS setup is the inclusion of a custom verification function for the X.509 certificate. This function ensures that the embedded attestation extension contains a valid attestation report.

The following Go snippet demonstrates how to configure TLS to perform this additional verification step.

	tlsConfig := &tls.Config{
		InsecureSkipVerify: true,
		RootCAs:            rootCAs,
		ServerName:         sni,
		VerifyPeerCertificate: func(rawCerts [][]byte, verifiedChains [][]*x509.Certificate) error {
			return atls.NewCertificateVerifier(rootCAs).VerifyPeerCertificate(rawCerts, verifiedChains, nonce)
		},
	}

Similar to the Attest() function, the behavior of the VerifyPeerCertificate() function, which veryfies the attestation extension, also depends on the underlying platform, such as AMD SEV-SNP or Intel TDX.

This approach relies on the assumption that the Agents private ephemeral key can't be stolen. Conclusion

As workloads move to untrusted environments like public clouds and edge platforms, securing communication is no longer just about encryption — it’s about verifiable trust. This is where RATS and aTLS come together.

RATS provides the architectural framework for generating, evaluating, and sharing attestation evidence, while aTLS seamlessly integrates this attestation into the TLS handshake. The result is not just secure communication, but communication where each party can prove its integrity and runtime state before exchanging sensitive data.

COCOS AI builds on this foundation by leveraging Confidential Computing and Trusted Execution Environments to enable secure multiparty computation. By adopting the RATS passport model and embedding attestation reports directly into X.509 certificates, COCOS AI ensures that every connection is both encrypted and verifiably trusted.

Future work includes implementing a different variety of TLS protocol in order to remove the flaw menitoned in this article.
